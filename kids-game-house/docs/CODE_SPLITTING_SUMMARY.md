# 📦 代码拆分优化总结

**版本**: v1.0  
**更新日期**: 2026-03-27  
**优化主题**: 控制单个文件代码量，按功能模块拆分

---

## 🎯 优化背景

在代码审查和实际开发中发现：

### 问题

- ❌ **单个文件过大**: PhaserGame.ts 超过 2000 行
- ❌ **职责不清晰**: 所有渲染、逻辑、碰撞检测都在一个类中
- ❌ **难以维护**: 修改一个地方可能影响其他地方
- ❌ **协作困难**: 多人修改同一文件容易冲突
- ❌ **新人恐惧**: 看到大文件不敢参与

### 影响

- 🔴 代码可读性差
- 🔴 Bug 难以定位和修复
- 🔴 单元测试难以编写
- 🔴 技术债务积累

---

## ✅ 优化方案

### 1. 制定代码量标准

| 类型 | 限制 | 说明 |
|------|------|------|
| **文件** | <600 行 | 超过需考虑拆分 |
| **函数** | <50 行 | 每个函数只做一件事 |
| **类** | <300 行 | 每个类只负责一个功能模块 |
| **方法** | <30 行 | 类中的方法应更精简 |

### 2. 推荐文件结构

```
src/components/game/
├── PhaserGame.ts (主框架，<600 行)
│   ├── 可复用框架层 (1-600 行)
│   └── 游戏特定层 (只保留编排调用)
│
├── components/ (组件库)
│   ├── GTRSLoader.ts
│   ├── ScreenAdapter.ts
│   ├── AudioManager.ts
│   └── ItemSystem.ts
│
├── renderers/ (🎨 渲染器目录 - 新增)
│   ├── PlayerRenderer.ts (<200 行)
│   ├── EnemyRenderer.ts (<200 行)
│   ├── BulletRenderer.ts (<150 行)
│   └── PowerupRenderer.ts (<150 行)
│
├── logic/ (🧠 游戏逻辑目录 - 新增)
│   ├── PlayerLogic.ts (<200 行)
│   ├── EnemyLogic.ts (<200 行)
│   ├── CollisionSystem.ts (<250 行)
│   └── ScoreSystem.ts (<100 行)
│
└── types/ (📝 类型定义目录)
    ├── player.types.ts
    ├── enemy.types.ts
    └── bullet.types.ts
```

### 3. 核心原则

- ✅ **单一职责**: 每个文件/类/函数只负责一件事
- ✅ **高内聚低耦合**: 相关功能放在一起，减少依赖
- ✅ **使用 index.ts**: 统一导出，简化导入路径
- ✅ **及时重构**: 发现代码集中立即拆分

## 🛠️ 实施指南

### 新项目

**从一开始就做好**:

1. **设计阶段**: 在 GDD 中规划文件结构
2. **开发阶段**: 严格遵守代码量限制
3. **评审阶段**: 检查是否符合拆分原则
4. **验收阶段**: 不符合标准的打回重做

### 已有项目

**渐进式重构**:

```
第 1 步：识别问题文件（找出>800 行的文件）
   ↓
第 2 步：优先级排序（从最乱的开始）
   ↓
第 3 步：小步重构（每次只重构一个模块）
   ↓
第 4 步：保证测试（重构前后功能一致）
   ↓
第 5 步：逐步推进（不要指望一步到位）
```

**重构顺序建议**:

1. 提取渲染逻辑 → `renderers/`
2. 提取游戏逻辑 → `logic/`
3. 提取类型定义 → `types/`
4. 创建 `index.ts` → 统一导出

## 🎯 判断标准

### 需要拆分的信号 🚨

- [ ] 单个文件超过 800 行
- [ ] 一个类有超过 10 个公共方法
- [ ] 一个函数超过 50 行
- [ ] 导入语句超过 20 个
- [ ] 文件名包含"And"（如`PlayerAndEnemyRenderer.ts`）
- [ ] 不同开发者频繁修改同一文件导致冲突

### 健康信号 ✅

- [x] 每个文件<600 行
- [x] 每个类<10 个公共方法
- [x] 每个函数<50 行
- [x] 导入语句<20 个
- [x] 文件名清晰表达单一职责
- [x] 可以轻松写出单元测试
- [x] 多人协作不冲突

## 📞 常见问题

### Q1: 拆分太细会不会增加复杂度？

**A**: 
- **短期看**: 似乎复杂了（文件多了）
- **长期看**: 简化了（每个文件都简单）

就像整理房间：
- ❌ 所有东西堆一起 → 看起来简单，找东西困难
- ✅ 分类收纳 → 看起来复杂，找东西容易

### Q2: 有些功能就是很复杂，无法拆分怎么办？

**A**: 继续拆分！再复杂的功能也可以分解为多个子功能。

**技巧**:
- 按数据流拆分（输入→处理→输出）
- 按业务逻辑拆分（玩家→敌人→道具）
- 按时间顺序拆分（初始化→更新→渲染）

### Q3: 如何说服团队成员遵守这个规范？

**A**: 用事实说话！

1. 展示对比：拿大文件和小文件对比给大家看
2. 分享痛苦：让大家说说修改大文件的痛苦经历
3. 制定规则：写入开发规范，Code Review 时检查
4. 工具辅助：使用 ESLint 等工具自动检测文件大小
