# 🎯 游戏设计先行 - 快速参考

**核心理念**: 先设计，后开发；不确认，不编码！

---

## 📋 一句话总结

在开始写代码之前，必须先完成**游戏设计文档**并通过**评审确认**。

---

## ⚡ 快速流程

```
1. 编写设计文档 → 2. 组织评审 → 3. 签字确认 ✅ → 4. 开始开发
```

**⚠️ 重要**: 没有签字确认就开始开发的，一律打回重做！

## 📝 GDD 核心章节

### 一、游戏概述（必填）

```markdown
- 游戏名称：
- 游戏类型：
- 目标用户：
- 核心玩法：（一句话描述）
- 游戏亮点：（1-3 个）
```

### 二、游戏对象设计（必填）

```markdown
玩家角色：
  - 移动方式：
  - 攻击方式：
  - 生命值：

敌对单位：
  - 类型 1：速度/血量/得分
  - 类型 2：速度/血量/得分

道具系统：
  - 道具 1：效果/持续时间
  - 道具 2：效果/持续时间
```

### 三、技术规格（必填）

```typescript
GRID_COLS = ___    // 横向格数
GRID_ROWS = ___    // 纵向格数
BASE_CELL_SIZE = __ px

数值设计：
  - 玩家速度：
  - 敌机生成间隔：
  - 子弹发射频率：
```

### 四、资源需求（必填）

```markdown
图片资源：
  - player.png
  - enemy1.png
  - ...

音频资源：
  - bgm.mp3
  - shoot.wav
  - ...
```

## 🔍 评审要点

### 趣味性（⭐⭐⭐⭐⭐）

- [ ] 游戏好玩吗？
- [ ] 操作简单吗？
- [ ] 难度合理吗？
- [ ] 有重复可玩性吗？

### 可行性（⭐⭐⭐⭐⭐）

- [ ] 技术能实现吗？
- [ ] 性能能达标吗？
- [ ] 时间够不够？

### 完整性（⭐⭐⭐⭐）

- [ ] 所有章节填了吗？
- [ ] 数据具体吗？
- [ ] 描述准确吗？

### 一致性（⭐⭐⭐⭐）

- [ ] 前后文一致吗？
- [ ] 数值统一吗？
- [ ] 风格匹配吗？

## 💡 成功秘诀

### 设计质量高

- 数据具体，不模糊
- 描述准确，不含糊
- 考虑周全，不遗漏

### 评审认真做

- 提前阅读文档
- 会上逐项检查
- 提出建设性意见
- 严格把关质量

### 执行不打折

- 需要修改走流程
- 完成后对照验收

## 📊 时间分配建议

```
设计阶段：20% （例如 6 天项目用 1-2 天）
开发阶段：70% （例如 6 天项目用 3-4 天）
测试阶段：10% （例如 6 天项目用 0.5-1 天）
```

**投资回报率**: 在设计上多花 1 小时，在开发中节省 5 小时返工

## 🆘 常见问题

### Q: 小项目也要这么复杂吗？

**A**: 要！大小项目都需要设计，这是保证质量的基础。

可以简化文档篇幅，但核心要素不能少：
- 游戏概述
- 对象设计
- 技术规格
- 评审确认

### Q: 一个人开发也需要评审吗？

**A**: 需要！可以：
- 找同事帮忙 review
- 隔天自己再审一遍
- 严格按检查清单逐项检查

### Q: 设计确认后完全不能改吗？

**A**: 可以改，但要：
1. 提出变更申请
2. 评估变更影响
3. 相关人员确认
4. 更新文档版本
5. 通知所有相关方

## 🎯 立即开始

### 第 1 步（今天）

1. 阅读 `DESIGN_FIRST_QUICKSTART.md`
2. 确定游戏创意
3. 编写概念设计书（1-2 页）

### 第 2 步（明天上午）

1. 使用 `GAME_DESIGN_TEMPLATE.md`
2. 编写详细 GDD
3. 自查所有检查项

### 第 3 步（明天下午）

1. 组织评审会议
2. 使用 `DESIGN_REVIEW_CHECKLIST.md`
3. 记录所有意见
4. 给出评审结论

### 第 4 步（获得确认后）

1. 所有相关方签字
2. 文档归档锁定
3. 开始开发！🚀
