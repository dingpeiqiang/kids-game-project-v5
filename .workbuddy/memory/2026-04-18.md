## space-invaders 资源管理器 - AI生成图片「应用到资源」修复

**问题**：resource-manager.html 里点击「应用到资源」按钮无效果。

**根因**：`vite-plugin-save-image.js` 中 `/__apply-resource` 和 `/__save-image` 两个路由在读取请求体时，用 `let body = ''; req.on('data', chunk => body += chunk)` 直接把 Buffer 字符串化拼接，遇到大 base64 图片（几十KB）会截断/乱码，导致 `JSON.parse` 失败，整个写文件操作失败。

**修复**：改用 `chunks: Buffer[]` 收集 + `Buffer.concat(chunks).toString('utf8')` 读取，封装为 `readBody(req)` 工具函数，两个路由都改用 async/await 调用。

**重要结论**：该项目完全用 Vite dev server（`npm run dev`），server.js 是遗留代码，不参与开发模式。所有 `/__save-image` / `/__apply-resource` 路由都应维护在 `vite-plugin-save-image.js` 中，server.js 里不需要也不应该重复实现。

## 纹理重复 + Failed to process file 修复（2026-04-18 第四轮）

**错误**：`Texture key already in use: bullet/ship/explosion` + `Failed to process file: image "enemy0..."`

**根因 1 - HEAD 预检误判**：Vite SPA fallback 会把任何 404 的静态资源路径返回 `index.html`（HTTP 200），导致 HEAD 预检 `resp.ok === true`，10 个不存在的图片被误判为"存在"并加入 `gtrsImageMap`，Phaser 真正加载时得到 HTML 内容 → `Failed to process file`。

**修复**：增加 `Content-Type` 检查，只接受 `image/*`，排除 HTML fallback：
```js
const ct = resp.headers.get('content-type') || '';
if (resp.ok && ct.startsWith('image/')) { gtrsImageMap[key] = img.src; }
```

**根因 2 - game 实例未销毁**：用户第二次点「开始游戏」时 `game` 实例已存在，纹理缓存仍有 bullet/ship/explosion 三个已生成的 key，再次 preload 时报 `Texture key already in use`。

**修复**：在 `initGame()` 里先 `game.destroy(true); await 100ms;` 再 `new Phaser.Game(config)`。

**根因 3 - 10 张图片文件不存在**：`assets/images/` 目录只有 3 个文件（bullet/ship/explosion），其余 10 个从未生成过。

**修复**：用 Sharp 程序化生成 10 张太空主题占位图（enemy0/1/2/boss/enemy_bullet/stars/particle/powerup_shield/powerup_rapid/powerup_life）。

## GTRS 图片被备用纹理覆盖修复（2026-04-18 第八轮）

**问题**：所有 13 张 GTRS 图片文件都存在且被正确加载（日志有 `📷 加载图片`），但游戏中显示的仍然是程序化生成的几何图形，不是真实图片。

**根因**：`preload()` 中 `generateFallbackTextures()` 在 `this.load.image()` 之后**同步**执行。此时 Phaser 的 image 加载队列尚未完成，`this.textures.exists(key)` 返回 false → `generateTexture()` 立即创建了同名纹理 → 之后队列完成时 Phaser 发现 key 已存在 → **静默跳过真实图片**（日志中之前看到的 `Texture key already in use` 就是这个原因）。

**修复**：将 `generateFallbackTextures()` 从 preload 同步代码移到 `this.load.on('complete', ...)` 回调中。这样 Phaser 先完成所有 image 加载并创建纹理，complete 之后再检查哪些 key 不存在，只对缺失的生成备用纹理。

**教训**：Phaser 的 `this.load.image()` 是异步的，只是提交到加载队列。在 preload 中不能同步检查 `textures.exists()`，因为纹理要到 `complete` 事件后才真正存在。

**问题**：资源管理器修改图片后，需要刷新游戏页面才能看到新资源。

**修复**：
1. `script-gtrs.js` 新增 `window.reloadGameTexture(key, base64DataUrl)` — 用 base64 图片直接替换 Phaser 纹理缓存，无需刷新页面
2. `resource-manager.html` 的 `apply()` 成功后调用 `window.reloadGameTexture()` 热替换纹理
