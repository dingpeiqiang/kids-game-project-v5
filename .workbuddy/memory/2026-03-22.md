## 修复 BasicInfoPanel 适用业务下拉框初始化只显示 value 不显示 name（两轮修复）

### 问题根因
`loadedOwnerType` 是单值 ref，`loadGameList` 和 `loadAppList` 都往同一个 ref 写，最后完成的那个会覆盖前一个的标记，导致 `applyThemeInfo` 里的就绪检查 `loadedOwnerType.value === 'GAME'` 失败（应用列表加载完后覆盖为 `'APPLICATION'`，游戏主题的就绪判断就永远是 false）。

同时原代码有两个 `onMounted`、`watch({ immediate: true })` 在列表未就绪时就触发赋值，三个问题叠加导致 el-select 找不到匹配 option，只能显示原始 value。

### 第三轮修复：采用一次性加载方案（最终方案）

**核心思路**：把"准备数据"和"渲染表单"彻底分离，在父组件层面控制渲染时机。

**改动文件**：
- `GTRSThemeCreatorV2.vue`（父组件）：
  - 新增 `dataReady` 布尔门控 + `initLoading` 状态
  - 新增 `gameList` / `appList` ref，在父组件统一加载
  - `onMounted` 改为 async：先 await 加载 gameList，再 await loadExistingTheme，完成后设 `dataReady = true`
  - 模板面板区域改为 `v-if="!dataReady"` 显示骨架屏，`v-else-if="editMode==='form'"` 显示表单（其中 BasicInfoPanel 增加 `:game-list` / `:app-list` prop）
- `BasicInfoPanel.vue`（子组件）：
  - 删除所有竞态代码：loadGameList、loadAppList、gameListLoaded、appListLoaded、watchEffect、isLoadingBusiness、gameApi import 全部删除
  - Props 新增 `gameList` / `appList`（由父组件传入）
  - `currentBusinessList` 直接用 computed 从 props 取，无需加载
  - `onMounted` 极简：直接 `formData.value = { ...props.modelValue.themeInfo }`
  - `watch(themeInfo)` 极简：直接赋值，无需任何判断

**结果**：BasicInfoPanel 挂载时，gameList 和 themeData 都已就绪，el-select 正确回显 label，彻底消除竞态。

## 修复 业务类型切换后 适用业务 下拉默认值显示数字（2026-03-22 14:40）

### 问题场景
业务类型（ownerType）切换后，`currentBusinessList` 切换到新列表，但当前 `ownerId` 若不在新列表中，el-select 无法匹配 label，只显示数字。

### 修复内容
**BasicInfoPanel.vue**：
1. 新增**兜底 option**：`v-if="formData.ownerId && !currentBusinessList.find(i => i.value === formData.ownerId)"` 显示 `ID: xxx（未知业务）`，防止显示裸数字
2. 新增 `watch(currentBusinessList)` + `validateOwnerIdInList()` 工具函数：列表更新时打印校验日志，方便调试 ownerId 命中状态
3. `onMounted` 挂载后调用 `validateOwnerIdInList('onMounted')` 校验一次

**GTRSThemeCreatorV2.vue**：
1. `loadExistingTheme` 加载主题时，**强制转换 ownerId 为数字类型**（`themeConfig.themeInfo.ownerId = Number(rawOwnerId)`），确保与 gameList 中的 value 类型一致
2. `gameList` 映射中处理 `gameId` 的类型：先取 `g.gameId ?? g.id`，再根据字符串/数字类型安全转数字，防止空值/非法字符导致 `NaN`
3. `onMounted` 改为 `Promise.allSettled` 并行加载 gameList + appList（appList 暂为空数组）
4. `appList` 明确为空数组（暂无后端 API），不再使用示例数据避免误导

### 实际问题定位（ID 665 显示未知业务）
用户反馈 ID 665 是贪吃蛇游戏，但下拉显示 `ID: 665（未知业务）`。根因是：
1. 后端 `configJson` 中的 `ownerId` 是字符串 `"665"`
2. gameList 中的 `value` 是数字 `665`
3. el-select 严格相等匹配：`"665" !== 665`，匹配失败触发兜底 option

**修复**：在 `loadExistingTheme` 加载后立即转数字，保证类型对齐。

## GTRS 规范设计决策：废弃 themeInfo 节点（2026-03-22 15:06）

### 设计讨论
用户最初考虑在 `themeInfo` 节点下使用 `theme_info` 表的真实字段名，但最终决定**废弃整个 `themeInfo` 节点**。

### 最终方案
- **configJson** 只存储：`specMeta` + `globalStyle` + `resources`
- **theme_info 表** 存储所有主题基础信息：`theme_id`, `author_id`, `theme_name`, `price`, `status` 等
- 后端 `ThemeController.downloadTheme()` 返回分离数据：
  - `themeInfo`: 来自数据库的完整主题信息对象
  - `config`: 只包含 GTRS 配置（specMeta + globalStyle + resources）

### 修改文件
- `gtrs-schema.json`: themeInfo 设为 `"type": "null"`，添加废弃注释
- `gtrs-template.json`: 删除 themeInfo 节点
- `gtrs-validator.ts`: GTRSTheme 接口移除 themeInfo，ThemeBasicInfo 改为独立接口（用于前端 UI 传递）
- 后端保持原设计不变（已实现分离返回）

### 优势
1. 数据不冗余：主题基础信息存在数据库，资源配置存在 configJson
2. 职责清晰：数据库管理主题元数据，JSON 管理资源和样式
3. 性能优化：下载主题时可以只返回需要的部分

### 上传数据格式
后端接收格式：
```typescript
{
  // theme_info 表字段
  themeName: string
  authorName: string
  price: number
  description: string
  thumbnail: string
  ownerType: 'GAME' | 'APPLICATION'
  ownerId: number
  status: 'pending'

  // GTRS 配置（specMeta + globalStyle + resources）
  configJson: string  // JSON 字符串
}
```

### 数据流
1. **加载主题**：后端返回 `{ themeInfo, config }` → 前端分别存入 `themeBasicInfo` 和 `themeData`
2. **编辑主题**：基本信息在 `BasicInfoPanel` 编辑（更新 `themeBasicInfo`），资源配置在其他面板编辑（更新 `themeData`）
3. **发布主题**：前端发送 `{ ...themeBasicInfo, configJson: JSON.stringify(themeData) }` → 后端分别存入 `theme_info` 表和 `config_json` 字段
