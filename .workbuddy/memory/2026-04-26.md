# 2026-04-26 工作日志

## simple-game 容器部署 CORS 403 问题修复

**问题**：`POST http://8.136.156.190:3001/api/user/register` 返回 403，错误信息 "Invalid CORS request"。

**根因**：`WebConfig.java` 中 `allowedOriginPatterns` 使用了 `"http://*"` 和 `"https://*"` 这两个写法，Spring 的 `allowedOriginPatterns` **不支持 `http://*` 这种通配形式**（只支持 `*` 或具体域名模式），导致所有来自 `8.136.156.190:3001` 的请求都被拒绝。同时设置了 `allowCredentials(true)`，与通配符冲突。

**修复**：将 `allowedOriginPatterns` 改为 `"*"`，移除 `allowCredentials(true)`（simple-game 使用 JWT Bearer Token，不依赖 Cookie，无需 credentials）。

**修改文件**：`kids-game-backend/kids-game-common/src/main/java/com/kidgame/common/config/WebConfig.java`

**结论**：后端需重新构建镜像并推送到服务器部署。

## simple-game Nginx 代理配置修复（第二轮）

**新问题**：后端日志完全没收到请求，说明 Nginx 代理本身有问题。

**排查发现 2 个 Nginx Bug**：
1. `proxy_set_header Content-Type $content_type;` — `$content_type` 在 Nginx 中不存在（应该是 `$http_content_type` 或直接不设置），导致 Content-Type 被清空，Spring Boot 无法解析 JSON 请求体
2. Nginx 的 `if ($request_method = 'OPTIONS')` 块里 `return 204` 后，非 OPTIONS 请求还是会走 CORS 双重 Header（Nginx + Spring 各加一次），引发浏览器拒绝

**修复**：重写 `docker/nginx/simple-game.conf`：
- 移除手动设置 Content-Type/Content-Length 的错误指令（让 Nginx 自动透传）
- 移除 Nginx 侧的 CORS 手动添加（由后端 Spring Boot 统一处理）
- 把 `location /` 放在 `location /api/` 后面

**同时修复**：`WebConfig.java` 中 `allowedOriginPatterns("http://*")` 改为 `allowedOriginPatterns("*")` 并移除 `allowCredentials(true)`

**结论**：需重建 simple-game 镜像（含新 Nginx conf）+ 后端镜像（含新 WebConfig）并重新部署。

## SecurityConfig CORS 修复（第三轮，真正根因）

**根因**：Spring Security 的 CORS 优先级**高于** WebConfig 的 `addCorsMappings`。`SecurityConfig.java` 里有独立的 `corsConfigurationSource()` Bean，白名单只有 `localhost` 和 `*.kids-game.com`，完全没有 `8.136.156.190`，所以 Spring Security 在过滤器层面就直接 403 了，请求根本进不到 Controller，后端日志也看不到。

**修复**：`SecurityConfig.corsConfigurationSource()` 改为 `allowedOriginPatterns("*")`，移除 `allowCredentials(true)` 和手动 `allowedHeaders` 列表（改为 `List.of("*")`）。

**教训**：Spring Boot 项目中如果同时有 `WebConfig.addCorsMappings` 和 `SecurityConfig.corsConfigurationSource`，Spring Security 的 CORS 过滤器先执行，WebConfig 的配置不生效。务必只在 SecurityConfig 里配置 CORS。
