## §0 使用本模板（策划引导摘要）

| 步骤 | 章节 | 合格标志 |
|------|------|----------|
| 1 定锚 | §1 | 竞品「抄/不抄」各 1 句；3 支柱有「若违反则…」 |
| 2 数值骨架 | §4 | **无空数字格**；与将来 `config.ts` 同名字段 |
| 3 反馈清单 | §5 | ≥8 个 `event_id`；轻/中/重不堆叠超标 |
| 4 首局剧本 | §3 | 每行有 `event_id` + 场上 kind |
| 5 指标与资产 | §2、§6、§7 | 每个 §2「目标值」是数字；§6 每行完整 EN Prompt |

**复制路径**：`docs/<gameId>-GDD.md`（勿直接改本模板文件）。

**密度底线**：§4 无空数值、§5≥8 事件、§3≥7 行；§2 每项有数字目标；对标 [市场参考 §3](GDD-MARKET-REFERENCES.md#3-跨类型高质量共性-p0-对标尺)，未达标文首标「未达 P0」。

| 交付等级 | 含义 | 最低要求 |
|----------|------|----------|
| **P0 可上线** | 能进儿童向合集商用 | §3 时间轴 + §4 全表有数 + §6 无「占位」占主视觉 + §9 P0 全过 |
| **P1 精品** | 留存与口碑 | P0 + §2 验收指标全达标 + §6 精灵表动效 + §5 音效齐全 |
| **原型** | 仅验证玩法 | 可仅 §4.1～4.3 + 程序 emoji 占位，**不可标商务定稿** |

## 附录 A：AI 硬性填写规约（禁止空话）

生成或补全 GDD 时 **必须** 遵守：

1. **禁止词**（出现则重写该格）：「适度」「丰富」「沉浸式」「高颜值」「体验良好」「灵活运用」「一定程度上」「等」「若干」「部分」。
2. **数值**：§4 每个 kind、波次、buff 必须有 hp/damage/speed/CD 中至少 3 项填数字；不得只写「中等血量」。
3. **时间**：§3 至少 7 行；§2 每个验收指标必须有目标值数字。
4. **反馈**：每个爽点必须在 §5 有 `event_id`；不得只写在 §2 形容词里。
5. **资产**：§6 每行必须有完整英文 Prompt（含 §7.2 master）；不得只写「可爱龙」。
6. **对标**：必填主参考产品 +「抄机制 X，不抄 Y」；节奏指标对齐 [GDD-MARKET-REFERENCES.md](GDD-MARKET-REFERENCES.md) §3，禁止只列游戏名。
7. **输出顺序**：先 §4 数值 → §5 事件 → §3 时间轴（引用 event）→ §6 资产 → §2 指标与数值对齐。
8. **自检**：填完后逐条执行附录 B；任一项否 → 在文首标注 **「未达 P0，仅原型」**。

**给 LLM 的一次性任务头**（复制后粘贴游戏名 + 类型）：

```text
你是儿童向手游主策划，产出「商务上线交付版」GDD（非概念稿）。
遵守文档附录 A；交付等级目标 P0。
输出完整 Markdown，章节 §1～§10 不省略；§4 可直接复制为 TypeScript 常量；§5 至少 8 个 event_id。
游戏：【名称】 gameId：【id】 type：【2d/3d】 主标签：【】
主参考：【市场小游戏名或玩法统称】 抄：【机制】 不抄：【】
先阅读 GDD-MARKET-REFERENCES 对应类型一节再写 §3/§2 指标。
```
