# 游戏开发规范 AGENTS.md

## 一、项目结构规范

### 1.0 去旧留新原则
- **新增功能优先**：开发新功能时，优先使用项目中已有的最新技术栈和架构模式。
- **旧代码清理**：修改或重构旧代码时，同步清理不再使用的废弃代码、注释和无用文件。
- **向后兼容**：在移除旧功能前，需确保所有依赖模块已迁移至新实现。
- **版本标记**：对即将废弃的接口或模块添加 `@deprecated` 注释，并指明替代方案。
- **定期清理**：每个迭代周期结束前，进行一次代码清理。

### 1.1 目录约定
```
games/<game-name>/
├── index.ts          # 游戏入口，暴露 init<GameName>() 函数
├── game.ts           # 主游戏类，游戏循环
├── config.ts         # 所有可配置常量
├── types.ts          # 所有类型/接口定义
├── logic/            # 纯逻辑模块（无渲染代码）
└── render/           # 纯渲染模块（无逻辑代码）
```

### 1.2 单一文件上限
- 单文件不超过 **500 行**（含空行/注释），超过必须拆分。
- `game.ts` 主文件应仅包含：类定义、游戏循环、事件绑定、模块编排。

## 二、代码组织规范

### 2.1 分层架构
- **编排层**：`game.ts` 主游戏类，负责循环、输入、模块调用
- **逻辑层**：`logic/` 纯函数，不操作 canvas
- **渲染层**：`render/` 纯绘图，不修改游戏状态
- **配置层**：`config.ts` 所有魔数、常量
- **类型层**：`types.ts` 所有接口定义

### 2.2 逻辑与渲染严格分离
- `logic/` 中的函数**严禁**调用 `ctx.draw*` 或任何 canvas 操作。
- `render/` 中的函数**严禁**修改游戏状态（如 `player.hp -= 1`）。

### 2.3 禁用 setTimeout 管理游戏流程
- 游戏内延迟逻辑应使用帧计数器或 `Date.now()` 差值判断。
- 仅在「游戏结束回主页」场景使用 `setTimeout`。

---

## 三、TypeScript 规范

### 3.1 所有数据必须定义类型
- 使用 `export interface` 定义实体类型。
- 禁止直接定义无类型对象字面量。

### 3.2 类型统一管理
- 跨模块使用的接口、类型别名统一放在 `types.ts`。
- 模块内部使用的工具类型可放在模块文件顶部。

### 3.3 函数签名明确
- 输入输出参数清晰，避免隐式依赖全局变量。

### 3.4 禁止 `any`
- 使用 `unknown` 替代 `any`，使用时必须类型断言。
- 使用 `as const` 定义常量枚举。

---

## 四、游戏架构规范

### 4.1 游戏循环
- 使用 `requestAnimationFrame` 驱动循环。
- 支持 delta time 计算。

### 4.2 游戏状态统一管理
- 所有运行时状态聚合到一个 `GameState` 对象中。
- 死亡/复活/过关时重置状态，不依赖重新实例化。

### 4.3 输入管理
- 使用即时输入状态对象而非事件驱动的标志位。

### 4.4 关卡配置化
- 所有关卡数据（敌人位置、平台布局、Boss配置）必须写在 `config.ts` 中。
- 禁止在代码中硬编码关卡逻辑判断。

---

## 五、性能优化规范

### 5.1 对象池与清理
- 离开屏幕的实体必须从数组中移除。
- 生命周期结束的粒子/特效及时清理。

### 5.2 渲染优化
- 只在 `requestAnimationFrame` 中渲染。
- 粒子数量控制上限（建议 ≤ 200）。
- `ctx.save()` / `ctx.restore()` 不嵌套超过 5 层。

### 5.3 碰撞优化
- 使用 AABB（轴对齐矩形包围盒）进行碰撞检测。
- 大量子弹碰撞时使用空间分区或先粗略过滤。

---

## 六、命名规范

### 6.1 文件/目录
- 全部小写，连字符分隔：`background.ts`, `game-loop.ts`
- 逻辑模块放 `logic/`，渲染模块放 `render/`

### 6.2 变量/函数
- 变量：camelCase，描述性名词：`playerHp`, `cameraX`
- 函数：camelCase，动词开头：`updatePlayer()`, `drawEnemies()`
- 布尔值：前缀 is/has/can：`isGrounded`, `hasBoss`
- 常量：PascalCase：`GAME_CONFIG`, `PLAYER_SPEED`

### 6.3 类/接口/类型
- 类：PascalCase：`ContraRpgGame`
- 接口：PascalCase，不加 I 前缀：`Player`
- 类型别名：PascalCase：`Level`

## 七、验收标准

### 7.1 编译检查
- `npx tsc --noEmit` 零 error。

### 7.2 代码质量检查表
- [ ] 单文件不超过 500 行
- [ ] 配置从 `config.ts` 读取而非硬编码
- [ ] 逻辑和渲染分层分离
- [ ] 类型有正确的 TypeScript 定义
- [ ] 无未使用的 import 或变量
- [ ] 粒子/实体清理逻辑完善

---

## 附：常见反模式

| 反模式 | 正确做法 |
|--------|---------|
| 单文件超 500 行 | 拆分为 logic/ + render/ 模块 |
| `setTimeout` 管理刷怪 | 使用时间戳差值判断 |
| 全局变量传递状态 | 通过函数参数传入 |
| 硬编码关卡判断 | 在 config.ts 中配置 |
| 坐标写死 | 使用 `GAME_CONFIG.CANVAS_WIDTH/HEIGHT` |
| 不清理离屏实体 | 每帧检查并移除超出范围的实体 |
