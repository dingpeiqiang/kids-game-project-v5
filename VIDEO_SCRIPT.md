```
**画面**: 对比新旧代码

```
现在分享一个重要的技术亮点：渐进式重构策略。

在升级食物系统时，我们面临一个挑战：
如何在保持向后兼容的前提下升级到新系统？

我们的解决方案非常巧妙：
```typescript
// 导入新系统但不强制替换
import { FoodType, createFood } from '../../types/FoodTypes'

// 复用现有类型定义（避免破坏）
type FoodType = 'normal' | 'bonus' | 'special'

// 在新代码中使用工厂函数
const newFood = createFood(position)

// 转换到旧接口（兼容）
this.currentFood = {
  x: newFood.position.x,
  y: newFood.position.y,
  type: newFood.type,
  score: newFood.score
}
```

这样做的好处是：
- 零破坏性变更
- 可以平稳过渡
- 随时可以回退
- 团队适应时间充足

这是一个非常实用的重构策略，
推荐大家在类似场景中使用。
```

#### 场景 8: 事件驱动架构 (7:00 - 8:00)

**画面**: 事件流程图

**旁白**:
```
事件驱动架构是我们的另一个亮点。

我们定义了完整的事件类型系统：
- 游戏状态事件：GAME_START, GAME_OVER
- 蛇相关事件：SNAKE_MOVE, SNAKE_EAT
- 食物相关事件：FOOD_SPAWN, FOOD_CONSUMED
- UI 相关事件：LOAD_PROGRESS, LEVEL_LOADED
- 目标相关事件：OBJECTIVE_UPDATED, OBJECTIVE_COMPLETE

所有组件都通过 EventBus 进行通信：
```typescript
// 单例模式
const eventBus = EventBus.getInstance()

// 发射事件
eventBus.emit({
  type: GameEventType.SCORE_CHANGED,
  payload: { score: 100 },
  timestamp: Date.now()
})

// 监听事件
eventBus.on(GameEventType.SCORE_CHANGED, (event) => {
  console.log('分数变化:', event.payload.score)
})
```

这种设计的优势非常明显：
- 完全解耦，组件互不依赖
- 易于调试，所有事件都有迹可循
- 灵活扩展，可以随时添加新的监听器
- 统一的通信机制，降低复杂度
